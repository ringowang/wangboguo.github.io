RESTful

什麼是REST呢？表象化狀態轉變Representational State Transfer，簡稱REST.

REST有主要有兩個核心精神：1. 使用Resource來當做識別的資源，也就是使用一個URL網址來代表一個Resource 2. 同一個Resource則可以有不同的Representations格式變化。

RESTful帶給Rails最大的好處是：它幫助我們用一種比較標準化的方式來命名跟組織Controllers和Actions。

为什么要使用Migration资料库迁移机制来定义资料库结构(Schema)?

  1. 讓資料庫的修改也可以納入版本控制系統，所有的變更都透過撰寫Migration檔案執行
  2. 方便應用程式更新升級，例如讓軟體從第三版更新到第五版，資料庫更新只需要執行rake db:migrate
  3. 跨資料庫通用，不需修改程式就可以用在SQLite3、MySQL、Postgres等不同資料庫

用MVC有什么好处

定义:M(Model)V(View)C(Controller),是软件开发的一种架构模式，目的是将不同的行为切开来，让代码撰写与维护上可以比较有规则。Rails开发是严格遵循MVC架构的。

  1. 界面与后台分离
  2. 维护相对简单,只要对应修正model就可以了
  3. 测试友好性(如果你需要写单元测试的话,MVC会很不错的)
  4. 可定制性(在MVC中基本上所有的东西都是可以定制的,对于某些定制性要求比较高的工作可以轻易胜任)
  5. 代码清晰(CodeBehind虽说分离的不错,但是还是有很多人直接在CodeBehind里面写SQL的)
  6. 轻量级,对于大型项目来说,可以大大提高加载速度(没有控件,没有视图状态,没有控件状态,页面加载速度快)
  7. 开源
  8. 强大的路由系统
  9. 对HTML代码和HTTP的严密控制力,MVC知道产生整洁、符合标准的标记的重要,它内置的HTML helper方法的输出完全符合标准

sdlong关于MVC的一些观点：

Model: 資料庫相關
Views: 前端網頁呈現
Controllers: 流程控制與運作

優點是:

  1. 可以把整個專案(Project)拆分
  2. 負責做前端的人只做 Views
  3. 寫流程控制的專心寫 Controllers
  4. 所有資料全部集中在 Models

版型(Layout)

Layout可以用來包裹樣板，讓不同樣板共用相同的HTML開頭和結尾部分。當Rails要顯示一個樣板給瀏覽器時，它會將樣板的HTML放到Layout的HTML之中。預設的Layout檔案是app/views/layouts/application.html.erb，其中yield就是會被替換成樣板的地方。所有的樣版預設都會套這個Layout。

好處是可以將網站的版型固定，只在需要出現內容的地方用 yield 引進來就可以了。

Helder

Helper 是一些使用在 Rails 的 View 當中，用 Ruby 產生/整理 HTML code 的一些小方法。通常被放在 app/helpers 下。

使用 Helper 封裝程式碼可以帶給專案以下一些優點：

  1. Don’t repeat yourself（DRY）程式碼不重複
  2. Good Encapsulation好的封裝性
  3. 提供 view 模板良好的組織
  4. 易於修改程式碼

局部樣板(Partial Template)

Partial 簡單說就是程式碼中的一小段，通常使用在 HTML 中讓 View的Code 可以更乾淨，將重複使用到的區塊切成獨立的 Partial，比方說頁首頁尾、表單、社群插件等等，讓任何一個頁面都可以讀取這段Partial而不用重複寫一次一模一樣的Code。

利用局部樣板(Partial)機制，我們可以將重複的樣板獨立出一個單獨的檔案，來讓其他樣板共享引用。比如_form.html.erb可以在new.html.erb和edit.html.erb里面通过render来引用。

SASS & SCSS

Haml 全名為 HTML Abstract Markup Language，它是提供網頁設計師撰寫 HTML 的另外一條途徑。

Sass 原先是附屬在 Haml 裡面的一個副功能。這也是 sass-convert 這個指令必須要安裝 haml 這個 gem 才能使用的原因。

Sass 的原理，是讓開發者可以透過「縮排」的方式去撰寫維護 CSS，同樣可以避免忘記關 TAG 而大爆炸的糗事。

Sass 原先使用的縮排，對於網頁設計師來說，還是相當不直觀。而且實務上也不能直接將舊有的 CSS 直接貼進 Sass 中。其實還是存在一定的不方便度。也因此 Sass 進行了進化，改良了 syntax，而 Sass 3 後來就被稱為 SCSS ( Sassy CSS)。

redirect_to & render

redirec_to通常使用在要讓使用者跳轉頁面的時候，會執行指定頁面的controller action。

而render則是將指定頁面的樣板拿出來而已，並沒有執行controller。

通常render的使用時機是讓使用者回到同一個頁面，例如表單填寫不完全時再重回表單填寫頁，這樣做的原因是render會傳模板給使用者，而這個模板在使用者第一次送出表單時就已經被存起來了，所以render同一個模板的時候就會保留剛剛使用者打的表單資料，不用全部重打。

相反的若使用redirect_to跳轉到同一個表單頁面就會是一個全新的模板，不會有任何送出前填寫的資料，所以適合在跳轉到不同頁面時使用。
